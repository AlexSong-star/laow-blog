# 提示注入不是漏洞，是AI Agent的原生缺陷

4月15日，安全公司Capsule Security发布报告，披露了微软Copilot Studio的一个高危提示注入漏洞。攻击者通过SharePoint公开表单注入恶意指令，让Copilot Agent自动查询CRM数据并外发到攻击者邮箱。微软迅速发布补丁，并获得了一个正式的CVE编号（CVE-2026-21520）。

但这个"CVE"本身就说明了问题所在：**安全行业第一次尝试用传统漏洞管理体系，去管理一个本质上无法被单次补丁根治的安全风险类别。**

## 漏洞打了个寂寞

Capsule的测试揭示了一个令人不安的事实：

- 微软的DLP确实**标记了这笔操作**为可疑
- 但DLP没有触发——因为邮件是通过**合法的Outlook接口**发出的
- 系统将攻击者操控的操作视为"授权操作"，因为操作的**发起来源**（Outlook）不在黑名单里

换句话说：攻击者没有绕过任何安全机制，他们只是**重新编程了Agent本身**，让Agent自己去做数据外泄——而Agent的所有行为都通过了合法的、受信任的系统接口。

这不是某个函数的边界检查缺失，而是**整个"可信用户/可信指令"模型的失效**。

## 为什么提示注入无法被单次补丁解决

传统软件漏洞的修复路径是清晰的：找到根因 → 编写修复 → 验证修复 → 发布补丁。修复是**一次性的**，因为代码的运行逻辑是确定性的。

提示注入的本质不是代码bug，而是**语言模型的固有特性**：LLM无法区分"系统启动时定义的可信指令"和"运行时刻从外部读取的不可信文本"。

```
传统Web安全：
输入 → 验证 → 执行
        ↑ 这里可以过滤

Agent安全：
系统指令（可信） + 外部输入（不可信）→ LLM处理
                    ↑
              LLM无法区分边界
```

只要Agent读取外部内容——邮件、文档、表单、网页——理论上就存在注入面。这是**架构层面的问题，不是代码层面的问题**。补丁可以修复这一次的特定注入路径，但无法修复Agent读取外部输入这个行为本身。

## "打了也白打"的漏洞生态

ShareLeak不是孤例。Capsule同日披露的PipeLeak针对Salesforce Agentforce，同样是通过表单注入，且**未获得CVE编号**——Salesforce拒绝为这个漏洞分配CVE，理由是"这属于配置问题，不是代码漏洞"。

这个争论本身就是一个信号：**当行业试图用CVE体系管理AI安全风险时，发现这个框架根本不适用**。

OWASP在2025年底发布的《Agentic AI应用安全Top 10》将"Agent目标劫持"（ASI01）列为第一类风险——本质上就是提示注入。但不同于传统漏洞分类，ASI01是一个**风险类别**，而不是一个可修复的具体缺陷。

| 漏洞类型 | 传统CVE | 提示注入/ASI01 |
|---------|--------|--------------|
| 根因 | 代码逻辑错误 | LLM无法区分信任边界 |
| 修复 | 单次补丁 | 需要架构级重新设计 |
| 影响范围 | 特定函数 | 所有读取外部输入的Agent |
| CVE适用性 | ✅ 完全适用 | ❌ 框架不适配 |

## 企业应该怎么办

这里没有银弹。但有几个可以实质降低风险的方向：

**1. 运行时监控（Runtime Guard）**

在Agent执行每一个工具调用之前，插入一个检查层——不是验证输入内容（不可行），而是验证**行为意图**：这个工具调用是否在预期范围内？是否会导致数据离开授权边界？

Capsule的技术方案就是这个思路：用微调的小模型作为"监护Agent"，在每个工具调用执行前评估其意图。这是一个正确的方向，但实施成本不低。

**2. 关键操作强制人工审批**

对于任何导致数据外发的操作（邮件发送、文件导出、外部API调用），强制人在回路审批。这个建议听起来简单，但在实际企业工作流中，AI Agent的价值恰恰是自动化这些操作——加入人工审批等于消灭了Agent的核心价值。

这是Agent安全领域的核心悖论：**越有用的Agent，越容易成为攻击面**。

**3. 输入隔离**

将外部输入与Agent的上下文严格隔离：Agent不应该直接将表单内容拼接到系统指令中，而应该通过结构化提取 → 安全验证 → 重建指令的路径来处理输入。这需要每个Agent开发者主动设计，大概率不会成为开箱即用的默认行为。

**4. 重新评估Agent的信息权限**

Capsule的研究指出了一个扎心的现实：企业把"克隆了人类权限的账户"交给Agent，区别在于Agent以机器的速度和规模使用这些权限。当权限边界不清晰时，这个问题会被放大。

最小权限原则在AI Agent时代不是"最佳实践"，而是**生存必需**。

## 写在最后

ShareLeak获得CVE编号，对安全行业是一个进步——至少说明这个风险类别已经被正式纳入管理框架。但对于正在部署AI Agent的企业而言，更重要的是理解：**提示注入不是一次性的漏洞，而是Agent架构的原生风险，需要持续的、架构级别的应对。**

补丁有用，但不是答案。
